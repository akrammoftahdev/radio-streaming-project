# Knowledge Base Update: May 24, 2026

## Overview of Today's Work
Today we focused on packaging the Next.js web application into a native Android Application using Trusted Web Activities (TWA) via Google's `bubblewrap` CLI. We successfully built and distributed the app as an APK (for sideloading) and an AAB (for the Google Play Store).

## Key Achievements & Hot Fixes
1. **Next.js PWA Configuration:**
   - Dynamically generated `manifest.webmanifest` using `src/app/manifest.ts`.
   - Set theme color to `#8B0000` (Dark Red), application name to `Radio Studio`, and package name to `com.onnet.studio`.

2. **Android App Generation (TWA):**
   - Installed `bubblewrap` CLI on the VPS (`195.35.48.184`).
   - Generated the Android project in `/root/android-app/`.
   - Created a secure `android.keystore` (Alias: `android`, Password: `password`) to sign the release APK and AAB.

3. **Digital Asset Links Verification:**
   - **Blocker:** The generated Android app displayed a browser URL bar at the top, ruining the native feel.
   - **Fix:** We extracted the SHA-256 fingerprint (`7D:8C:85:E7:8B:0F:70:C2:10:C0:13:14:4B:3E:FD:06:C6:7A:D5:5B:8B:E6:26:7A:20:A7:30:75:27:F0:B6:33`) from the `android.keystore` and placed it inside a new file `assetlinks.json`.
   - Hosted `assetlinks.json` inside the `.well-known` directory on our Next.js frontend to prove domain ownership to Android.

4. **Middleware Blocking Static Assets:**
   - **Blocker:** The Next.js `proxy.ts` middleware was intercepting requests to `.apk`, `.aab`, `manifest.webmanifest`, and `.well-known` files and redirecting them to the `/login` page because they were unauthenticated.
   - **Fix:** We updated the `matcher` array in `frontend/src/proxy.ts` to explicitly exclude these public static assets:
     `matcher: ["/((?!api|uploads|_next/static|_next/image|favicon.ico|manifest.webmanifest|.well-known|icons|RadioStudio\\.apk|RadioStudio\\.aab).*)"]`
