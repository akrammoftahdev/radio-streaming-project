## 1. Studio V3 Modular Refactor (ATTEMPTED — REVERTED)

### Goal
Modularize the monolithic `studio-ui-v2.tsx` (3,193 lines) into 5 custom hooks + a lean component `studio-ui-v3.tsx` (1,594 lines).

### Hooks Created
| Hook | File | Responsibility |
|---|---|---|
| `useAudioMixer` | `src/hooks/studio/useAudioMixer.ts` | AudioContext, mixer dest, mic, bg, gain nodes, monitor, VU meter |
| `useDsp` | `src/hooks/studio/useDsp.ts` | Full DSP chain, presets, bypass, per-filter toggles |
| `useRecording` | `src/hooks/studio/useRecording.ts` | Session recording start/stop, upload |
| `useSfxPads` | `src/hooks/studio/useSfxPads.ts` | SFX buffer decode, trigger, stop, volume |
| `useQueuePlayer` | `src/hooks/studio/useQueuePlayer.ts` | Queue playback, crossfade, auto-advance, seek |

### Outcome: FAILED
- TypeScript compiled with zero errors
- Production build passed locally and on VPS
- **Runtime behavior was broken** — the user reported "studio is destroyed"
- Root cause: hooks sharing refs behave differently than inline refs in a monolithic component. Cross-hook ref timing (e.g., `audioCtxRef` created in `useAudioMixer` but needed synchronously in `useQueuePlayer`) caused silent failures.

### Lesson Learned
> **Never deploy a refactor to production without thorough runtime testing.** TypeScript compilation success does NOT guarantee runtime correctness when splitting a monolithic component into hooks that share AudioContext refs, GainNodes, and MediaElementAudioSourceNodes.

### Files Created (still on disk, NOT active)
- `frontend/src/app/studio/studio-ui-v3.tsx` — 1,594 lines

## 2. VPS Deployment Cascade Failure

### What Happened (chronological)
1. **`git reset --hard origin/main`** on VPS — replaced the single-app frontend repo with the full monorepo structure, destroying the working file layout
2. **Build failures** — TypeScript picked up `prisma.config.ts`, `backend-audio/`, and other monorepo files, causing TS errors
3. **Orphan process** — a manual `npx next start` test left an orphan process on port 3000, causing PM2 to crash-loop with `EADDRINUSE`
4. **PM2 crash-loop** — 81 restarts. Even after fixing the build, PM2 couldn't start because port 3000 was occupied
5. **Multiple failed fix attempts** — each SSH reconnection attempt was rate-limited by fail2ban

### Recovery Steps
1. `fuser -k 3000/tcp` — killed orphan process
2. `git reset --hard c2f2ce1` — restored original "Create Next App" commit
3. `mv prisma.config.ts prisma.config.ts.bak` — removed file that blocked builds
4. `rsync` all source files from local → VPS (bypassing git entirely)
5. `npm install && npm run build && pm2 start npm --name frontend -- start`

### Lessons Learned
> 1. **NEVER use `git reset --hard` on a production VPS.** The VPS has a different repo structure than the monorepo.
> 2. **Always use `rsync` for deploying to VPS** — it's safer than git operations.
> 3. **Always kill orphan processes before restarting PM2** — `fuser -k 3000/tcp`.
> 4. **Rate-limit SSH connections** — fail2ban blocks rapid reconnections. Use a single long-running SSH session.
> 5. **Test locally before deploying** — especially for refactors.

## 3. Fader Unit Mismatch Bug (FIXED)

### Problem
The background and queue volume faders used a slider range of `0-100` (percentage display), but the `GainNode.gain.value` was set directly to the slider value without dividing by 100.

| Fader | Default Value | What GainNode Got | Expected |
|---|---|---|---|
| Background (`bgVolume`) | `0.5` | `0.5` (nearly silent) | `0.5` (50%) |
| Queue (`queueVolume`) | `0.8` | `0.8` (nearly silent) | `0.8` (80%) |
| User moves to 75 | `75` | `75` (75× louder than max!) | `0.75` |

### Fix
- Changed defaults: `bgVolume: 0.5 → 50`, `queueVolume: 0.8 → 80`
- Added `/ 100` at every point where the value goes to a `GainNode.gain.value` or `fadeGain()` call
- **10 locations fixed** in `studio-ui-v2.tsx`:
  - Initial bg gain setup (line 740)
  - `applyBgGain` function (line 772)
  - Queue resume (line 1033)
  - Queue fade-in (line 1174)
  - Crossfade B fade-in (line 1247)
  - Queue volume sync effect (lines 1362, 1364)
  - Queue-ended bg restore (line 1152)
  - Crossfade-chain bg restore (line 1293)
  - Queue-stop bg restore (line 1394)
  - BG reconnect after mic switch (line 2065)
  - BG wire on ws.onopen (line 2211)

### Impact
This bug was the reason the queue audio got "fucked up" when the user moved faders. Setting queue volume to e.g. 75 sent `75.0` to `GainNode.gain.value`, which is 75× the maximum volume, causing severe distortion/clipping.

## 4. Multi-File Upload Fix (FIXED)

### Problem
The file picker `<input type="file">` in the media library tabs (خلفية, أغاني, فواصل, إعلانات) was missing the `multiple` attribute. Users had to select files one at a time.

### Fix
Added `multiple` to all 4 file inputs:
```html
<input type="file" accept="audio/*" multiple className="hidden" ... />
```

### Note
The handler function `handleLocalFilePick` already supported `FileList` (used `Array.from(files).forEach()`), so the only change needed was the HTML attribute.

## 5. DSP Routing Clarification

### Confirmed Architecture
DSP effects apply to the **microphone only**, not to background or queue audio.

```
micSource → [HP → LP → EQ → Comp → Gate → DeEsser → Reverb → Delay → Warmth → Limiter] → micGain → mixer
bgSource  → bgGain  → mixer  (no DSP)
queueSrc  → queueGain → mixer  (no DSP)
sfxSource → sfxGain → mixer  (no DSP)
```

This is intentional — DSP effects (compressor, noise gate, de-esser, reverb) are voice processing tools meant for live microphone input, not for pre-mixed music tracks.
