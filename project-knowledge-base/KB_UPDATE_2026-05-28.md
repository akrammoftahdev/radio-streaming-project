## Overview
~3-hour debugging session to fix **recording playback** in the mobile app's Home/WaitScreen. The root cause was a **one-line auth mismatch** ÔÇö the mobile play endpoint only accepted JWT via `?token=` query parameter, while the mobile app's HTTP client (axios) sends JWT via `Authorization: Bearer` header. Multiple wrong approaches were attempted before identifying the real issue.

## 1. The Bug: Mobile Recording Playback Doesn't Work

**Symptom:** User taps a recording in the RecordingMiniPlayer Ôćĺ player bar appears Ôćĺ no audio plays, no seek control, duration shows 0:00. No error messages in console.

**Root Cause (found after ~3 hours):** Auth method mismatch.
- The `/api/mobile/recordings/play/[filename]` route ONLY checked `url.searchParams.get("token")` for JWT authentication.
- The mobile app's `api` module (axios) sends the JWT via `Authorization: Bearer <token>` header, NOT as a query parameter.
- Result: Every play request returned HTTP 401, but the error was silently swallowed by various audio APIs.

**Fix:** 5 lines added to `play/[filename]/route.ts`:
```typescript
// ÔöÇÔöÇ 1. Auth: JWT from query string OR Authorization header ÔöÇÔöÇÔöÇÔöÇÔöÇÔöÇÔöÇÔöÇÔöÇÔöÇÔöÇÔöÇÔöÇÔöÇÔöÇ
let token = url.searchParams.get("token");
if (!token) {
  const authHeader = req.headers.get("authorization");
  if (authHeader?.startsWith("Bearer ")) {
    token = authHeader.slice(7);
  }
}
```

**File:** `frontend/src/app/api/mobile/recordings/play/[filename]/route.ts`

## 2. Failed Approaches (In Order) ÔÇö DOCUMENTED FOR FUTURE REFERENCE

### ÔŁî Attempt 1: expo-audio `useAudioPlayer` with plain string URL
**What:** Passed `playbackUrl` string directly to `useAudioPlayer(item.playbackUrl)`.
**Why it failed:** Silent failure ÔÇö no error, no network request. Initially thought the source format was wrong (string vs `{ uri: string }`), but both formats are valid per expo-audio types: `AudioSource = string | number | null | { uri?: string }`. The real issue was the 401 being silently ignored by the native audio player.
**Time wasted:** ~30 min

### ÔŁî Attempt 2: expo-audio with `{ uri: string }` format
**What:** Changed to `useAudioPlayer({ uri: item.playbackUrl })`.
**Why it failed:** Same silent 401 rejection. Changing the source format had no effect because the URL itself returned 401.
**Time wasted:** ~15 min

### ÔŁî Attempt 3: WebView + HTML5 `<audio>` element
**What:** Complete rewrite using a WebView containing an HTML page with `<audio>` element, communicating via `postMessage`. Logic: if it works in the web studio browser, it should work in a WebView.
**Why it failed:** `MEDIA_ERR_SRC_NOT_SUPPORTED` (code 4). The WebView loaded inline HTML (`source={{ html }}`), and the `<audio>` element couldn't load the external URL. Even with `baseUrl: 'https://studio.egonair.com'` set, the audio request failed.
**Time wasted:** ~20 min

### ÔŁî Attempt 4: WebView + fetch() test before audio.src
**What:** Added a `fetch(url, { method: 'HEAD' })` test inside the WebView to check the URL before loading it into the audio element.
**Why it failed:** CORS error ÔÇö `fetch()` from inline HTML (`about:blank` origin) is blocked by CORS. Error message: "ě«ěĚěú ě┤ěĘ┘âěę: load failed".
**Key Learning:** `fetch()` and `XHR` from inline WebView HTML are subject to CORS. `<audio>` and `<img>` elements bypass CORS (same-origin policy doesn't apply to media elements), but the audio element also failed due to the underlying 401.
**Time wasted:** ~15 min

### ÔŁî Attempt 5: WebView + XHR blob download
**What:** Used `XMLHttpRequest` with `responseType: 'blob'` to download the audio file inside the WebView, then `URL.createObjectURL(blob)` to play it.
**Why it failed:** Same CORS issue ÔÇö XHR from inline HTML is blocked. Error: "XHR failed".
**Time wasted:** ~10 min

### ÔŁî Attempt 6: FileSystem.downloadAsync
**What:** Used `expo-file-system` `downloadAsync(playbackUrl, localFile)` to download the MP3, then play the local file with `expo-audio`.
**Why it failed:** TWO issues:
1. **Deprecation warnings** ÔÇö expo-file-system v54 deprecated `makeDirectoryAsync`, `getInfoAsync`, `downloadAsync` in favor of new `File` and `Directory` classes. Fixed by importing from `expo-file-system/legacy`.
2. **Download hangs forever** ÔÇö `downloadAsync` never completed. The `playbackUrl` included `?token=JWT` but `downloadAsync` apparently couldn't reach the server. Added a 15-second timeout ÔÇö confirmed the download times out.
**Time wasted:** ~25 min

### ÔŁî Attempt 7: axios download Ôćĺ HTTP 401 ÔćÉ ROOT CAUSE FOUND
**What:** Used the app's existing `api.get()` (axios with Bearer auth interceptor) to download the file, then save with `FileSystem.writeAsStringAsync`.
**What happened:** `HTTP 401` error clearly logged in console.
**Why:** The `api` module sends `Authorization: Bearer <token>` header. The play route ONLY checks `url.searchParams.get("token")`. Bearer header was ignored Ôćĺ 401.
**Resolution:** Added Bearer header check to the play route (5 lines). Audio played immediately after.
**Time to final fix once root cause found:** ~5 min

## 7. Known Constraints & Gotchas (New)

1. **expo-audio `useAudioPlayer` silently fails on 401** ÔÇö if the remote URL returns an HTTP error, the player shows duration 0 and never plays. No error is thrown or logged.
2. **WebView inline HTML (`source={{ html }}`) cannot fetch external URLs** ÔÇö XHR, fetch, and sometimes even `<audio>`/`<img>` elements fail due to CORS/origin restrictions.
3. **`FileSystem.downloadAsync` may hang on certain servers** ÔÇö unclear why, but it never completes for the play endpoint. Using axios + writeAsStringAsync is a reliable alternative.
4. **expo-file-system v54 deprecated legacy API** ÔÇö use `import * as FileSystem from 'expo-file-system/legacy'` to suppress warnings.
5. **Arraybuffer to base64 conversion in React Native** ÔÇö `btoa()` works on Hermes engine for converting Uint8Array bytes to base64 string for `writeAsStringAsync`.
6. **VPS has NO fail2ban** ÔÇö SSH timeouts are network issues, not bans. Don't waste time on whitelisting.
7. **Next.js production mode REQUIRES `npm run build`** ÔÇö editing source files on VPS without rebuilding has NO effect.
