# Knowledge Base Update (2026-05-31)

## Session Summary: Admin Management System & Infrastructure Fixes

### Good Stuff & Accomplishments
1. **Admin CRUD Completion**: Successfully completed the Create, Update, and Delete flows for System Admins. Added full profile fields (Name, Email, Phone, Profile Photo).
2. **Next.js 15 Compliance**: Refactored page parameters in `admin/admins/page.tsx` to properly await `searchParams` as a Promise, aligning with Next.js 15 server component requirements.
3. **i18n Localization Sweep**: Successfully spawned an `i18n_converter_v2` subagent that swept through the `admin/` and `station-manager/` directories to remove hardcoded Arabic text and replace them with `next-intl` translation keys in both `en.json` and `ar.json`.
4. **Native File Uploads via Server Actions**: Successfully integrated native `<input type="file">` components directly into standard Next.js form server actions (`createAdmin`, `updateAdmin`) to process multipart form-data natively without the need for external client-side upload wrapper components.

### Bad Stuff, Blockers & Problems
1. **Deployment IP & Path Mismatch**: Previous agent sessions were failing deployments by targeting an outdated IP address (`194.163.143.146`) and incorrect file paths.
2. **Client-Side Routing Glitches**: The Edit and Delete side panels in the Admin dashboard were glitching because standard HTML `<a>` tags were used for cancel buttons instead of Next.js `<Link>` components, causing full page reloads and search parameter drops.
3. **Avatar Upload Context Mismatch**: The existing `AvatarUpload` React component was hardcoded to use the active `session.user.id`. When an Admin tried to use it to upload a photo for *another* Admin, it was overwriting the active Admin's own profile photo.

### How We Fixed Them
1. **Infrastructure Hardcoding**: Explicitly mapped deployment scripts to target `root@195.35.48.184:/var/www/egonair/` using `rsync -arvzR`, followed by `npm run build && pm2 restart all` via SSH.
2. **Component Upgrades**: Replaced all `<a>` tags with `import Link from 'next/link'` in the Admin page to preserve the client-side router cache and search params.
3. **Server Action File Processing**: Bypassed the `AvatarUpload` component entirely for Admin creation/editing. Instead, added native file processing inside `actions.ts` that saves the incoming `File` object directly to `public/uploads/avatars/` with a unique UUID, and creates/updates the target user's `PresenterProfile` record in Prisma.
