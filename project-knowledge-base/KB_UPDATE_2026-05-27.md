# Knowledge Base Update: May 27, 2026

## Overview
Full day spent fixing **media file playback** (queue songs, background music) in the mobile app's `AVAudioEngine` pipeline. Multiple interrelated bugs were discovered and resolved involving the `scheduleFile` completion callback, format converter lifecycle, Arabic filename handling, keepalive interference, and mixer format adaptation.

---

## 1. Bugs Found & Fixed

### 🐛 BUG: `scheduleFile` Completion Fires Too Early (Queue Songs Stop After ~1 Second)
**Symptom:** Queue songs played for ~1 second then went silent.  
**Root Cause:** `AVAudioPlayerNode.scheduleFile(_:at:completionHandler:)` default completion type is `.dataConsumed`, which fires when the audio data is **buffered** (consumed by the player's internal buffer), NOT when it finishes **playing**. For most files, this fires within ~1 second of scheduling.  
**Fix:** Changed to `.dataPlayedBack` completion type:
```swift
// BEFORE (broken):
player.scheduleFile(audioFile, at: nil) { [weak self] in ...

// AFTER (fixed):
player.scheduleFile(audioFile, at: nil, completionCallbackType: .dataPlayedBack) { [weak self] _ in ...
```
**Key Learning:** `.dataConsumed` ≠ "playback finished". Always use `.dataPlayedBack` when you need to know when audio actually finishes playing through the output.  
**File:** `LiveAudioStreamModule.swift` → `scheduleAndPlay()`

---

### 🐛 BUG: Mic Distortion When Opening Mic While Music Plays (Format Converter Stale)
**Symptom:** Opening mic while a queue song or background music was playing caused severe audio distortion on the SHOUTcast stream.  
**Root Cause:** TWO interrelated issues:

1. **Captured converter variable:** The `installTap` closure captured the `converter` variable by value (Swift closure capture). When `playFile()` updated `self.audioConverter`, the tap closure still used the **old** captured converter with the wrong format → garbage output.

2. **Fixed tap format:** The tap was installed with a specific `tapSourceFormat`. When the mixer reconfigured (e.g., player reconnected with a stereo file, or mic toggled), the tap continued receiving buffers in the old format.

**Fix (3-part):**

**Part A — Dynamic tap format:** Install tap with `nil` format so it delivers buffers in the mixer's CURRENT output format, adapting automatically:
```swift
// BEFORE:
mainMixer.installTap(onBus: 0, bufferSize: bufferFrames, format: tapSourceFormat) { ... }

// AFTER:
mainMixer.installTap(onBus: 0, bufferSize: bufferFrames, format: nil) { ... }
```

**Part B — Dynamic converter recreation in tap callback:** Check buffer format on every tap call and recreate converter if it changed:
```swift
mainMixer.installTap(onBus: 0, bufferSize: bufferFrames, format: nil) { [weak self] (buffer, _) in
    guard let self = self, self.isStreaming else { return }
    
    let bufferFormat = buffer.format
    if self.audioConverter == nil || self.audioConverter!.inputFormat != bufferFormat {
        self.audioConverter = AVAudioConverter(from: bufferFormat, to: tapTargetFormat)
    }
    guard let converter = self.audioConverter else { return }
    // ... convert and send
}
```

**Part C — Invalidate converter on mic toggle:** When `setMicVolume` changes, the mixer may silently reconfigure its output format. Force converter recreation:
```swift
Function("setMicVolume") { (volume: Float) in
    self.micGainNode?.outputVolume = max(0.0, min(1.0, volume))
    self.audioConverter = nil  // Force tap to recreate converter
}
```

**Key Learning:** 
- Swift closures capture variables by value at creation time — use `self.property` for mutable state in long-lived closures
- `AVAudioMixerNode` can silently change its output format when inputs are added/removed/reconnected
- Always install taps with `nil` format for maximum robustness
- Invalidate the converter whenever the audio graph topology changes

**Files:** `LiveAudioStreamModule.swift` → `buildEngineGraph()`, `playFile()`, `setMicVolume()`

### 🐛 BUG: Arabic Filenames Fail to Load
**Symptom:** Audio files with Arabic characters in the filename (e.g., `أغنية.mp3`) silently failed to play.  
**Root Cause:** `URL(string:)` rejects non-ASCII characters. The file path from JS contained unencoded Arabic characters.  
**Fix:** Multi-step URL resolution with percent-encoding fallback:
```swift
// 1. Try as-is (works for already-encoded file:// URIs)
fileURL = URL(string: urlString)
// 2. Try percent-encoding for Arabic/Unicode
if fileURL == nil, let encoded = urlString.addingPercentEncoding(withAllowedCharacters: .urlQueryAllowed) {
    fileURL = URL(string: encoded)
}
// 3. Try as raw file path
if fileURL == nil || !FileManager.default.fileExists(atPath: fileURL!.path) {
    fileURL = resolveFileURL(urlString)  // includes percent-decoding fallback
}
```
**File:** `LiveAudioStreamModule.swift` → `playFile()`

### 🐛 BUG: Keepalive Silence Packets Interfere With Engine Audio
**Symptom:** Audio had brief cuts/dropouts every 2-3 seconds when playing music without mic.  
**Root Cause:** When mic was closed, `closeMic()` started a keepalive interval that sent base64 silence packets every 5 seconds via WebSocket. But the `AVAudioEngine` tap was ALSO sending real PCM audio via `onAudioData`. The server received BOTH streams interleaved → audio corruption.  
**Fix:** Keepalive only runs BEFORE the engine starts (during the connect → first-use gap). Once `ensureEngine()` starts the engine, keepalive is stopped. The engine tap sends continuous PCM (silence when idle, mixed audio when playing), keeping the server alive without separate keepalive packets.  
**File:** `[stationId].tsx` → `ensureEngine()`, `closeMic()`, `ws.onopen`

### 🐛 BUG: Queue Songs Don't Play Without Opening Mic First
**Symptom:** Adding a song to queue and pressing play did nothing — SHOUTcast stayed silent.  
**Root Cause:** The `AVAudioEngine` only started when the mic was opened (via `startMic()`). Without opening mic first, there was no engine to play files through.  
**Fix:** Introduced `ensureEngine()` — a lazy engine starter called before ANY audio action (play music, open mic). It requests mic permission (required for `engine.inputNode`) and starts the engine on first use:
```javascript
const ensureEngine = useCallback(async () => {
    if (engineRunningRef.current) return true;
    const { granted } = await requestRecordingPermissionsAsync();
    if (!granted) return false;
    // Start engine, subscribe to events, set volumes...
    engineRunningRef.current = true;
    return true;
}, [mediaVolume]);
```
Called in: `openMic()`, `handleSetBg()`, `playNow()`  
**File:** `[stationId].tsx`

### 🐛 BUG: Queue Song Plays at Volume 0 (fadeMediaVolume Race Condition)
**Symptom:** Queue song appeared to play (UI showed PLAYING) but SHOUTcast was silent.  
**Root Cause:** `playNow()` called `fadeMediaVolume(0, BG_FADE_OUT_SEC)` to fade out background, then 100ms later called `playFile()` + `setMediaVolume()`. But the fade ramp (2-3 seconds) overrode the `setMediaVolume` call, keeping volume at 0.  
**Fix:** Removed fade and setTimeout from `playNow()`. Direct stop + play:
```javascript
LiveAudioStream.stopFile();
LiveAudioStream.playFile(item.file.uri, false);
LiveAudioStream.setMediaVolume(mediaVolume);
```
**File:** `[stationId].tsx` → `playNow()`

## 2. Updated Architecture: Engine Lifecycle

### Before (May 26)
```
Connect → keepalive
Open mic → startMic() → engine starts
Close mic → stopMic() → engine stops → keepalive resumes
```
**Problem:** Engine only runs while mic is open. No media playback without mic.

### After (May 27)
```
Connect → keepalive silence
First audio action (mic OR music) → ensureEngine() → mic permission → engine starts → keepalive stops
Close mic → setMicVolume(0) — engine stays running
Disconnect → stopAll() → engine stops
```
**Benefit:** Engine runs from first use until disconnect. Media plays independently of mic state.

## 3. Updated Data Flow (May 27)

```
iPhone Mic Hardware (AVAudioSession .playAndRecord)
  ↓ engine.inputNode → micGainNode (volume 0-1) ──→ mainMixerNode
  
Audio Files (background/queue)
  ↓ AVAudioPlayerNode → mediaGainNode (volume 0-1) → mainMixerNode

mainMixerNode ──→ installTap(format: nil) → dynamic AVAudioConverter → Int16 PCM
  ↓ base64 → onAudioData event → JS → WebSocket → backend → SHOUTcast

mainMixerNode ──→ monitorGainNode (volume 0-1) → outputNode (DJ headphones)
```

### Key Design Decisions
| Decision | Rationale |
|---|---|
| Tap format = `nil` | Adapts to mixer format changes (mic toggle, file format switch) |
| Dynamic converter | Recreated when buffer format changes — no stale converters |
| Converter invalidated on mic toggle | Mixer silently reconfigures when inputs change |
| `.dataPlayedBack` completion | Fires when audio FINISHES playing, not when buffered |
| `ensureEngine()` lazy start | Engine starts on first use, works with or without mic |
| No keepalive when engine running | Engine tap sends continuous PCM, keepalive would corrupt stream |
| Mic close = volume 0, not stop | Engine keeps running for media playback after mic closes |

## 4. Ducking Behavior (Updated)

Ducking now applies to **both** background music and queue songs:

| Scenario | Media Volume |
|---|---|
| Background playing, mic OFF | `bgVolume` (default 0.5) |
| Background playing, mic ON | `bgVolume × BG_DUCK_RATIO` (≈7.5%) |
| Queue song playing, mic OFF | `mediaVolume` (default 0.7) |
| Queue song playing, mic ON | `mediaVolume × BG_DUCK_RATIO` (≈10.5%) |
| Mic closes | Restore to pre-duck volume |

## 6. Known Constraints & Gotchas

1. **`AVAudioEngine.inputNode` requires mic permission** — even if mic is muted, accessing `inputNode` in `buildEngineGraph()` requires the permission to be granted first
2. **`scheduleFile` does NOT loop natively** — must manually re-schedule in completion handler with `audioFile.framePosition = 0`
3. **`URL(string:)` rejects Arabic/Unicode** — always use percent-encoding fallback for user-provided filenames
4. **Swift closure captures are by value** — mutable state in long-lived closures (like `installTap`) must use `self.property`, not captured local variables
5. **`devicectl` auto-launch frequently fails** on Xcode 16 + iOS 16 — app installs fine but must be launched manually on device
6. **Hot reload (shake→reload) doesn't reset native engine state** — always force-quit the app after native (Swift) changes
