## Overview
Two full days (May 25–26) were spent getting the **Native Mobile App microphone** to successfully stream live audio from iPhone → WebSocket → backend FFmpeg → SHOUTcast. ## 1. Mobile App Audio Streaming Pipeline — FINAL Working Architecture

### ✅ CURRENT PRODUCTION APPROACH: Custom AVAudioEngine Native Module

The app uses a **custom Expo native module** (`LiveAudioStream`) written in Swift. It uses Apple's `AVAudioEngine` with `installTap(onBus:)` to capture continuous PCM buffers directly from the hardware microphone — no files, no stops, no gaps, no clicks.

### Data Flow (confirmed working, perfect audio quality)
```
iPhone Mic Hardware (AVAudioSession .playAndRecord)
  ↓  AVAudioEngine.inputNode.installTap(onBus: 0) → ~100ms PCM buffers
  ↓  AVAudioConverter (inputFormat → Int16 44100Hz mono)
  ↓  Data → base64EncodedString() → sendEvent("onAudioData", { data, size })
  ↓  JS: LiveAudioStream.onAudioData(event => ws.send(event.data))
  ↓  WebSocket URL: wss://studio.egonair.com/audio?token=XXX&format=pcm
  ↓  backend-audio FFmpeg: -f s16le -ar 44100 -ac 1 -i pipe:0 → libmp3lame → pipe:1
  ↓  SHOUTcast SOURCE stream (MP3)
```

### Key Advantages Over Previous expo-audio Approach
| Aspect | expo-audio (old) | AVAudioEngine (current) |
|---|---|---|
| Streaming | Stop/start every 2s, file-based | Continuous, never stops |
| Audio quality | Clicks/pops at chunk boundaries | Perfect, gapless |
| File I/O | Read WAV file, strip header, delete | None — direct memory buffers |
| Dependencies | expo-audio, expo-file-system/legacy, buffer | Only ExpoModulesCore (Swift) |
| Future mixing | Impossible | Built-in mixer node support |
| Background audio | Not supported | AVAudioSession keeps app alive |

### Module Location
```
mobile-app/modules/live-audio-stream/
├── expo-module.config.json          # Expo autolinking config
├── ios/
│   ├── LiveAudioStream.podspec      # CocoaPods spec (required for autolinking)
│   └── LiveAudioStreamModule.swift  # AVAudioEngine native code (~90 lines)
└── src/
    └── index.ts                     # TypeScript wrapper: start(), stop(), onAudioData()
```

### JS API
```typescript
import LiveAudioStream from '../../../modules/live-audio-stream/src';

// Start streaming
LiveAudioStream.start({ sampleRate: 44100, channelCount: 1 });

// Listen for continuous PCM chunks (~100ms each)
const sub = LiveAudioStream.onAudioData((event) => {
  ws.send(event.data); // base64-encoded Int16 PCM
});

// Stop streaming
LiveAudioStream.stop();
sub.remove();
```

## 2. Failed Approaches — DO NOT RETRY

### 2.1 expo-audio useAudioRecorder (SUPERSEDED)
- **What it did:** Recorded WAV files every 2 seconds, read file, stripped 44-byte header, sent base64 PCM.
- **Problems:** 2-second stop/start gaps causing audible clicks/pops. File I/O overhead. Race conditions between file read and recorder restart.
- **Status:** Fully replaced by AVAudioEngine native module. All expo-audio recorder code removed from `[stationId].tsx`.

### 2.2 react-native-live-audio-stream (FAILED on iOS 18 + Expo New Arch)
- **Package:** `react-native-live-audio-stream` — streams continuous raw PCM chunks via native AudioQueue.
- **Result:** `AudioConverterNew returned -50` — iOS AudioConverter fails to create converter from 0 channels to 1 channel.
- **Root cause:** The package's native iOS code does not properly initialize AudioQueue input format under Expo's Fabric/Bridgeless new architecture (iOS 18 simulator).
- **Status:** Installed in `package.json` and `Podfile.lock` but NOT used. Can be removed in cleanup.

### 2.3 expo-audio RecordingPresets.HIGH_QUALITY (M4A — FORMAT MISMATCH)
- **What happened:** Default preset records M4A/AAC. Server expects WebM or raw PCM.
- **Result:** FFmpeg received M4A container chunks with headers — could not parse as s16le or WebM. Silent failure, stale timeout.

### 2.4 Spreading RecordingPresets (NATIVE CRASH)
- **What happened:** `useAudioRecorder({ ...RecordingPresets.HIGH_QUALITY })` — spreading the preset object causes a native SharedObject crash in expo-audio.

## 3. Expo SDK 54 Breaking Changes Discovered

### 3.1 expo-file-system: readAsStringAsync DEPRECATED
- **Symptom:** `Method readAsStringAsync imported from "expo-file-system" is deprecated` — throws an error, does NOT just warn.
- **Fix:** Import from `expo-file-system/legacy`.
- **Note:** No longer needed for audio streaming (native module eliminated file I/O), but still relevant for other features.

### 3.2 SafeAreaView Deprecated from React Native Core
- **Fix:** Use `import { SafeAreaView } from "react-native-safe-area-context";` instead.

### 3.3 expo-audio setAudioModeAsync Property Names
- **Correct property:** `allowsRecording: true` (NOT `allowsRecordingIOS`).
- **Note:** No longer used directly — AVAudioEngine sets its own AVAudioSession.

## 4. Backend Audio Server — Pre-existing format=pcm Support

### Discovery
The `backend-audio/src/index.ts` server ALREADY had full support for raw PCM input via the `format` query parameter on the WebSocket URL:
- `?format=webm` (default) → FFmpeg uses `-f webm` input format
- `?format=pcm` → FFmpeg uses `-f s16le -ar 44100 -ac 1` input format

### Stale Session Timeout
- **STALE_TIMEOUT_MS = 15,000 (15 seconds)** — if no binary/text audio chunk is received, the server closes the WebSocket with reason `stale_timeout`.
- **Solution:** Send pre-computed base64 silence (`'A'.repeat(5460) + 'AA=='`) every 5 seconds while connected but not yet recording. Keepalive stops when mic activates.

## 7. Expo Local Module Autolinking — Requirements Discovered

For a local Expo module in `modules/` to be autolinked:
1. **`expo-module.config.json`** at module root — defines platform and Swift class name
2. **`.podspec` file in `ios/` directory** — REQUIRED for CocoaPods to compile the Swift code
3. **NO `package.json` needed** inside local modules (confuses autolinking)
4. After adding a new module: `pod install` in `ios/` THEN `npx expo run:ios`
5. Verify detection: `npx expo-modules-autolinking resolve 2>&1 | grep ModuleName`
